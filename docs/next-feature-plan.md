# Project Veil 下一阶段功能清单

## 当前项目判断

- 当前最稳定的可玩链路已经成型：探索、采集、占矿、招募、PVE 战斗、PVP 遭遇、账号持久化都能在 H5 调试壳里跑通。
- `packages/shared` 和 `apps/server` 的玩法底层已经比表现层更完整，真正的短板主要在主客户端 Cocos 的“操作闭环”和“产品化表现”。
- 本轮用 `develop-web-game` 自动化脚本实测了 3 条主链路：
  - 战斗流：击败守军后获得 `gold +300`，英雄升到 `Lv 2`。
  - 资源流：拾取木材、占矿、推进天数后，木材来到 `15`。
  - 招募流：回招募所补兵后，英雄部队数来到 `16`，金币剩余 `260`。

## 旧版候选功能点

1. `Cocos 装备背包与战利品闭环`
   - 把已有的装备、背包、掉落数据真正接到 Cocos 主客户端里，支持查看战利品、穿戴/卸下装备、即时看到属性变化。
   - 原因：shared/H5 已经具备这条链路，但 Cocos 目前更多只是展示装备摘要。

2. `战报 / 回放中心`
   - 把服务端已保存的 `battle replays`、近期战斗摘要和奖励记录接到 H5/Cocos 账号面板，形成可回看的战报中心。
   - 原因：后端数据已经落地，但当前玩家侧几乎感知不到，价值没有被承接。

3. `Cocos 战斗表现正式化`
   - 替换占位式战斗转场，补齐受击、技能释放、胜利/失败的正式动画、音效和资源绑定。
   - 原因：现在主循环是通的，但主客户端仍偏“能跑的壳”，产品感和可展示性不够。

4. `多人对抗体验收口`
   - 围绕英雄遭遇 PVP，补齐更明确的进入战斗提示、战后结算、房间状态反馈和对手信息展示。
   - 原因：联机底座和双人遭遇已经能用，下一步应该把“能打”做成“可持续联调、可持续验证”的正式体验。

5. `第二批内容包`
   - 新增至少一套地图布局，以及 2 到 3 组新的建筑/中立怪/掉落组合，让探索和养成更有差异。
   - 原因：核心系统并不少，但当前单局内容密度仍偏薄，影响后续留存验证。

## 旧版排序建议

- 第一优先：`Cocos 装备背包与战利品闭环`
- 第二优先：`Cocos 战斗表现正式化`
- 第三优先：`战报 / 回放中心`
- 第四优先：`多人对抗体验收口`
- 第五优先：`第二批内容包`

## 2026-04 游戏向路线刷新

### 当前新判断

- 现在仓库最不缺的是“系统名词”，而是“把已有系统编排成一个更像游戏的前 15 到 30 分钟体验”。
- `campaign / daily quest / battle pass / replay / progression / 多套地图包` 已经都在仓库里有数据或 UI 壳，下一步应该优先打通玩家真正能感知到的目标感、奖励感和重复游玩价值。
- 因此，下一阶段不建议继续优先扩更多外围系统，而应该优先做 `新手首日体验 -> PVE 内容密度 -> 长期成长闭环 -> Cocos 主客户端产品化体验`。

### 建议版本顺序

1. `V0.6 首局到首日体验`
   - 目标：让新玩家在 10 到 20 分钟内始终知道“下一步做什么”，并且在首战后感知到明确成长反馈。
2. `V0.7 PVE 内容密度与长期成长`
   - 目标：让第二局、第三局不只是重复一张图，而是开始出现章节推进、任务追逐和成长积累。
3. `V0.8 Cocos 主客户端产品化`
   - 目标：让主客户端从“主流程可跑”提升到“适合扩大试玩和留存验证”。

### V0.7 首个落地切片

- `Cocos Lobby PVE 前台入口`
  - 把已有 `campaign` 与 `daily dungeon` 数据从“只能进面板找”推进到“Lobby 一眼可见、能直接接着打”的前台入口。
  - 目标：玩家回到 Lobby 时，能直接看到当前主线下一步和今日 PVE 内容，而不是只看到一组系统按钮。
  - 最小验收：
    - Lobby 明确展示 `下一主线任务` 与 `今日每日地城`
    - 战役 / 地城入口在 lobby 态点击后，能把玩家带进正确的房间内 panel，而不是停留在无反馈状态
    - 账号态下会主动同步每日地城摘要，避免入口卡片长期显示空白

## V0.6 可直接开工的 5 条

1. `新手引导状态机接线`
   - 把首次移动、首次采集、首次建筑交互、首次战斗真正串成步骤流。
   - 验收：新玩家 5 分钟内始终有明确下一步提示。

2. `首章主线任务落地`
   - 把 campaign 面板从“查看任务”推进到“任务真正驱动地图目标、对话和奖励”。
   - 验收：首章至少有一条完整的接任务 -> 完成任务闭环。

3. `首战结算页产品化`
   - 把首战后的奖励、升级、掉落和下一步建议统一到一个清晰页面。
   - 验收：玩家打完第一场战斗后能在一个面板里看懂“赢了什么、接下来去哪”。

4. `首日任务奖励闭环`
   - 把 daily quest / progression 接到首局核心行为，让玩家前 15 分钟内自然完成 2 到 3 个小目标。
   - 验收：完成行为后立刻可领取奖励，不需要自己找入口。

5. `新手漏斗埋点与平衡校准`
   - 围绕 `进房 -> 首次移动 -> 首次采集 -> 首次招募 -> 首次战斗胜利 -> 首次领奖` 建立漏斗。
   - 验收：能明确看到流失主要发生在哪一步，并能支撑下一轮平衡调整。

## 推荐执行顺序

- 第一批：`新手引导状态机接线`
- 第二批：`首章主线任务落地`
- 第三批：`首战结算页产品化`
- 第四批：`首日任务奖励闭环`
- 第五批：`新手漏斗埋点与平衡校准`

## 最小可验证版本

- 如果下一轮只做一个最小版本来验证留存，建议只做 `1 + 2 + 3`。
- 原因：这三条做完，项目就会第一次具备一个更完整的“前 15 分钟体验”，足以开始验证玩家是否理解目标、是否愿意继续玩，而不是只验证系统是否存在。

## 2026-04-16 同步仓库后的调整

### 新观察

- 教程不是空白能力，`apps/cocos-client/assets/scripts/VeilRoot.ts` 已经存在 `1/3 -> 3/3` 的账号级引导状态机和 `tutorial_step` analytics。
- 主线任务也不是从零开始，`tests/e2e/campaign-mission-flow.spec.ts` 已经覆盖 `mission start -> reward settlement -> unlock progression -> analytics`。
- 首日活动/奖励链路已明显前进，`tests/e2e/seasonal-event-lifecycle.spec.ts` 已经覆盖 `progress -> reward claim -> leaderboard -> analytics`。
- 因此，`V0.6` 的重点应从“先补底层状态机”改成“把教程、主线、首日奖励真正绑成一个首局闭环”。

### 调整后的 V0.6 优先级

1. `教程与首章主线联动`
   - 不再把“新手引导状态机接线”当成从零建设，而是把已有教程步骤真正绑定到首章 mission flow。
   - 验收：玩家完成教程推进时，首章目标会同步驱动地图上的下一步行为。

2. `首战结算页产品化`
   - 仍然保持高优，因为这是当前最直接影响玩家“打完第一场后是否理解成长”的体验点。
   - 验收：奖励、升级、掉落、下一步建议统一到一个结果页里。

3. `首日任务 / 活动奖励曝光收口`
   - 基于已有的 daily / seasonal / progression 数据与 E2E，把前 15 分钟内能完成的奖励明确推到前台，而不是藏在多个面板里。
   - 验收：玩家在首局内能自然看到并领取至少一条任务或活动奖励。

4. `新手漏斗补齐与平衡校准`
   - 在 tutorial / mission / seasonal analytics 已有基础上，补齐首局关键节点漏斗。
   - 验收：能清楚看到 `进房 -> 引导推进 -> 首战 -> 首次领奖` 的主要流失节点。

### 调整后的最小可验证版本

- 如果下一轮只做一个最小版本来验证留存，建议改成 `教程与首章主线联动 + 首战结算页产品化 + 首日任务/活动奖励曝光收口`。
- 这版比旧方案更贴近同步后的仓库现实，因为它利用的是已经存在的 tutorial、campaign、seasonal-event 基础，而不是重复建设已经落下来的系统。
